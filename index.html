1. Introduction

This section is non-normative.

Responsive Web Components need to respond to content box size changes. An example is an Element that displays a map:

  • it displays a map by tiling its content box with Element tiles.

  • when resized, it must redo the tiling.

Responsive Web Applications can already respond to viewport size changes. This is done with CSS media queries, or window.resize event.

The ResizeObserver API is an interface for observing changes to Element's clientWidth and clientHeight. It is an Element counterpart to window.resize event.

clientWidth and clientHeight are observed because they are the best match for size of the Element's content box.

Desirable side effects of watching clientWidth are:

  • observation will fire when watched element is inserted/removed from DOM.

  • observation will fire when watched element display is hidden.

  • observations do not fire for inline elements.

  • observations will not be triggered by CSS transforms.

offsetWidth was also considered. It was rejected because it contained width of the scroller. Scroller width is dependent upon Element’s children, and this can cause notification loops.

var ro = new ResizeObserver( entries => {
  for (entry of entries) {
    console.log(
        'element:', entry.target,
        'width: ', entry.clientWidth,
        ' height: ', entry.clientHeight);
  }
});

let el = document.createElement('div');
document.body.appendChild(el);
ro.observe(el);
el.appendChild(document.createElement('input'));
// el.clientWidth changes after el.appendChild

el = document.createElement('div');
ro.observe(el);
document.body.appendChild(el);
// el.clientWidht changes after el is inserted into body.

el.style.display = 'none';
// el.clientWidth changes after el was hidden.

// All these size changes would trigger ResizeObserver

2. Resize Observer API

It is modeled after MutationObserver and IntersectionObserver.

3.2. Algorithms

3.2.1. Gather notifications entries

It computes all pending resize notifications for a document. To gather notification entries#gather-notification-entriesReferenced in:3.4.1. HTML Processing Model: Event Loop (2), run these steps:

  1. For each observer in resizeObservers run these steps:

    1. Clear observer’s activeEntries.

    2. For each observation in observer.observationTargets run these steps:

      1. Let sizeChanged be the result of calling observation.observeSize.

      2. If sizeChanged is true then add new ResizeObserverEntry(target) to observer.activeEntries.

3.2.2. Has notification entries

To determine if Document has notification entries#has-notification-entriesReferenced in:3.4.1. HTML Processing Model: Event Loop (2) run these steps:

  1. Set hasEntries to false.

  2. For each observer in {Document/resizeObservers}} run these steps:

    1. If observer.activeEntries is not empty, set hasEntries to true.

  3. return hasEntries.

3.2.3. Broadcast Resize notifications

To broadcast resize notifications#broadcast-resize-notificationsReferenced in:2.2. ResizeObserverCallback3.4.1. HTML Processing Model: Event Loop for a document, run these steps:

  1. For each observer in document.resizeObservers run these steps:

    1. If observeractiveEntries slot is empty, continue.

    2. For each entry in activeEntries

      1. Set entry.clientWidth slot to entry.target.clientWidth.

      2. Set entry.clientHeight slot to entry.target.clientHeight.

    3. Invoke observer.callback with activeEntries.

    4. Clear observer.activeEntries.

3.2.4. Deliver Resize Limit Error

To deliver resize limit error notification#deliver-resize-limit-error-notificationReferenced in:3.4.1. HTML Processing Model: Event Loop run these steps:

  1. For each observer in resizeObservers do this:

    1. Clear observer.ResizeObserver.activeEntries

  2. Create a new ErrorEvent

  3. Initialize event’s message slot to "ResizeObserver loop limit exceeded.".

  4. Dispach the event to document’s window.

3.3. ResizeObserver Lifetime

A ResizeObserver will remain alive until both of these conditions are met:

  • there are no scripting references to the observer.

  • the observer is not observing any targets.

3.4. External Spec Integrations

3.4.1. HTML Processing Model: Event Loop

ResizeObserver processing happens inside the step 7.12 of the HTML Processing Model event loop.

Step 12 is currently underspecified as:

For each fully active Document in docs, update the rendering or user interface of that Document and its browsing context to reflect the current state..

Existing step 12 can be fully specified as:

For each fully active Document in docs, run the following steps for that Document and its browsing contents:

  1. recalc styles

  2. update layout

  3. paint

ResizeObserver extends step 12 with resize notifications. It tries to deliver all pending notifications by looping until no pending notifications are available. Infinite looping is prevented by limiting the number of times loop is repeated to RESIZE_OBSERVER_LOOP_LIMIT constant. Looping too many times generates an error.

Note: RESIZE_OBSERVER_LOOP_LIMIT is currently arbitrarily set to 16. It might get adjusted based on data we obtain from real-world usage.

Step 12 with ResizeObserver notifications is:

For each fully active Document in docs, run the following steps for that Document and its browsing contentx:

  1. recalc styles

  2. update layout

  3. let resizeNotificationCount be 0

  4. gather notification entries for Document

  5. repeat while (document has notification entries && resizeNotificationCount < RESIZE_OBSERVER_LOOP_LIMIT)

    1. increment resizeNotificationCount by 1

    2. broadcast resize notifications

    3. recalc styles

    4. update layout

    5. gather notification entries for Document

    6. if (document has notification entries && resizeNotificationCount is RESIZE_OBSERVER_LOOP_LIMIT) then deliver resize limit error notification
